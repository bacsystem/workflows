# Fase 4a — Quick Fixes: Integration Branch Creation and Duplicate Skill

**Date:** 2026-07-17
**Status:** Approved

## 1. Context

Found during the cys independence-proof smoke test (2026-07-17, real plugin
installed from GitHub):

1. **Missing integration branch creation.** Both `commands/flow.md` and
   `commands/run-plan.md` assume `integrationBranch` already exists — neither
   creates it. `flow.md`'s step 7 only handles the case where it *already*
   exists ("ask whether to continue on it or pick another name"); nothing
   creates it when it doesn't. Same gap in `run-plan.md`'s step 5→6. When the
   branch is missing, the first task's merge agent fails with `git
   merge-base --is-ancestor` reporting "not a valid object name" — a
   confusing failure disguised as a merge conflict, discovered live during
   the smoke test's first run of the Subtract API pilot.

2. **Duplicate skill registration.** After installing the cys plugin,
   `cys:parallel-plan-executor` appears twice, identically, in the available
   skills listing. Root cause (confirmed via documentation research):
   `workflows/` is not one of Claude Code's documented plugin
   auto-discovery directories (`skills/`, `commands/`, `agents/`, `bin/`,
   etc.), but the platform appears to scan it anyway for any `.js` file
   containing the `Workflow` tool's `export const meta = {...}` header —
   and registers each match. `workflows/` currently contains two such files:
   `parallel-plan-executor.template.js` (hand-edited source) and
   `parallel-plan-executor.js` (generated by `scripts/build-workflow.js`),
   both carrying the identical `meta` block, so both get registered.

## 2. Fix 1 — Create the integration branch before launching

In both `commands/flow.md` and `commands/run-plan.md`, insert a step
immediately before the "Launch the Workflow tool" step:

- Check whether `integrationBranch` exists locally (`git -C <repo-path>
  show-ref --verify --quiet refs/heads/<integrationBranch>`).
- If it does **not** exist: create it from `develop`
  (`git -C <repo-path> branch <integrationBranch> develop`) — no
  confirmation needed, this mirrors what the command already told the user
  it would do when it suggested the branch name.
- If it **does** exist: unchanged — the existing "continue on it or pick
  another name" question still applies.

No engine change needed — this is a command-prompt-only fix, since the
branch must exist before `parallel-plan-executor`'s first merge agent runs.

## 3. Fix 2 — Stop the duplicate skill registration

Move `workflows/parallel-plan-executor.template.js` out of `workflows/`
into a new `workflows-src/` directory, so `workflows/` contains **only**
the generated `parallel-plan-executor.js` — the sole file with a `meta`
block in any directory Claude Code might scan.

Changes required:

- `workflows-src/parallel-plan-executor.template.js` (moved, content
  unchanged).
- `scripts/build-workflow.js`: update its template input path from
  `workflows/parallel-plan-executor.template.js` to
  `workflows-src/parallel-plan-executor.template.js`; output path
  (`workflows/parallel-plan-executor.js`) unchanged.
- `CLAUDE.md`, `README.md`, `README.es.md`: update every reference to
  `workflows/parallel-plan-executor.template.js` to
  `workflows-src/parallel-plan-executor.template.js`.
- `tests/build-workflow.test.js`: no path assertions reference the
  template's location directly (it only reads the *generated* file's
  output), so no change expected there — verify during implementation.

## 4. Testing / verification

- Fix 1: no automated test possible for command `.md` files (no test
  surface); verified by re-running the smoke test's exact failure scenario
  manually — launch against a repo where `integrationBranch` does not yet
  exist, confirm it gets created and the run's first merge succeeds.
- Fix 2: `npm run build` must still produce an identical
  `workflows/parallel-plan-executor.js` (byte-for-byte, verified with
  `git diff` showing no unexpected changes) after moving the template.
  `npm test` must stay green. Real verification of the actual fix requires
  a fresh `/plugin install` + `/reload-plugins` cycle (interactive, outside
  automated test scope) confirming `cys:parallel-plan-executor` now
  appears exactly once.

## 5. Out of scope

- Investigating *why* Claude Code scans `workflows/` at all, or filing an
  upstream bug report — this spec only covers the repo-side mitigation.
- Any change to the `Workflow` tool's own named-workflow resolution
  (`.claude/workflows/`) — unrelated to this plugin's directory layout.
